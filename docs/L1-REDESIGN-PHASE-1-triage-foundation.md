## 1. What this phase delivers

After Phase 1 ships, every Jira ticket that fires the L1 Triage webhook will:

1. **Have its Jira priority automatically set from the SIEM severity field** (`customfield_10038`). Today the priority is whatever the Logic App / Service Desk created the ticket with (often left as "Medium"). After Phase 1, priority always matches severity within ~40s.
2. **Be auto-assigned to a GSOC service account** (new env var `JIRA_GSOC_ACCOUNT_ID`). If the env var is empty, this step is skipped with a log line ‚Äî same defensive pattern as today's L1/L2 fields.
3. **Get a second priority adjustment from an AI L1 Triage call.** The LLM reads the ticket (summary + description + entity IOCs + severity) and decides whether the impact warrants a different priority than severity alone implies. Examples:
   - Severity "Low" but description says "Domain controller dumped" ‚Üí escalate to Highest
   - Severity "High" but description says "Test alert generated by analyst" ‚Üí de-escalate
   - LLM confidence below threshold (< 0.7) ‚Üí keep severity-based priority
4. **End up with one of three explicit verdict labels** ‚Äî replaces today's generic `IOC_Detection` / `investigating`:
   - `True-Positive` ‚Äî at least one IOC was flagged by VT, AbuseIPDB, or SOCRadar
   - `Benign-Positive` ‚Äî at least one reputation source returned data; nothing flagged
   - `Unknown` ‚Äî no IOCs found OR all reputation sources returned no data

Everything that works today keeps working: IOC extraction from typed entity fields, regex fallback, reputation lookups, the L2 reassignment, the enrichment comment posted to Jira, the dedup pipeline. Phase 1 is additive plus the label rename.

## 3. Severity ‚Üí Priority mapping

Phase 1 ships a hardcoded mapping that handles both the standard severity-tier names and the SCDM "Sev-X" convention seen in `customfield_10038`:

| Input (any case) | Jira priority |
|---|---|
| `Critical`, `Sev-0` | `Highest` |
| `High`, `Sev-1` | `High` |
| `Medium`, `Sev-2` | `Medium` |
| `Low`, `Sev-3` | `Low` |
| `Informational`, `Lowest` | `Lowest` |
| anything else | (no change ‚Äî logged as `priority sync skipped: unknown severity`) |

The LLM Triage call (step 3 above) can override this baseline per-ticket if its confidence is ‚â• 0.7.

## 5. Test plan for the team

After deploy, validate by creating test tickets in the SCDM Jira project (whitelisted in `JIRA_ENRICHMENT_PROJECT`).

| # | Scenario | Setup | Expected result |
|---|---|---|---|
| 1 | Severity sync ‚Äî Critical | Create test ticket; set `customfield_10038` (Severity) = "Critical" (or "Sev-0") | Jira priority auto-updates to "Highest" within ~40s of creation (after stabilization sleep) |
| 2 | Severity sync ‚Äî High / Medium / Low / Informational | Repeat #1 for each severity value | Priority becomes High / Medium / Low / Lowest respectively |
| 3 | GSOC assign (env var set) | Set `JIRA_GSOC_ACCOUNT_ID` to a valid Jira account; create any test ticket | Assignee field shows the GSOC account within ~40s |
| 4 | GSOC assign (env var empty) | Unset `JIRA_GSOC_ACCOUNT_ID`; create test ticket | Assignee unchanged; webhook log shows "GSOC assign skipped ‚Äî JIRA_GSOC_ACCOUNT_ID not set" |
| 5 | LLM Triage override UP | Create ticket with `customfield_10038` = "Low" but description like "Domain controller compromised; admin account dumped" | Priority initially Low, then escalated to High/Highest by LLM call. Webhook log shows LLM rationale. |
| 6 | LLM Triage override DOWN | Create ticket with severity "High" but description like "Test alert generated by analyst" | Priority initially High, then de-escalated by LLM. Log shows rationale. |
| 7 | LLM Triage no-override | Create ticket where LLM agrees with baseline | Priority stays at baseline. Log shows LLM agreed (or confidence < threshold). |
| 8 | Verdict label ‚Äî True-Positive | Inject ticket with a known-malicious IOC (e.g. a sinkholed C2 IP that VT flags) | Ticket ends up labelled `True-Positive`. Comment header: `VERDICT: TRUE-POSITIVE`. |
| 9 | Verdict label ‚Äî Benign-Positive | Ticket with clean IOCs (e.g. `8.8.8.8`, `google.com`) | Labelled `Benign-Positive`. Comment header: `VERDICT: BENIGN-POSITIVE`. |
| 10 | Verdict label ‚Äî Unknown | Ticket with no IOCs (or IOCs that all three reputation sources fail to score) | Labelled `Unknown`. Comment header: `VERDICT: UNKNOWN`. |
| 11 | LLM Triage failure resilience | (For SOC team to skip unless instructed) Temporarily set bad Azure OpenAI endpoint to force LLM call to fail | Pipeline continues; severity-mapped priority stays in effect; enrichment still runs; comment still posts. No webhook failure. |
| 12 | End-to-end smoke | Standard Sentinel-originated ticket | All of: priority set, GSOC assigned (if env configured), LLM rationale logged, enrichment comment posted, correct verdict label |

Phase 1 is **signed off** when scenarios 1‚Äì10 and 12 all pass. Scenario 11 is optional.

### Where to look in logs

Open Azure Log Analytics (workspace `a5be1877-ff08-4a4f-8fba-9b3e0664ded4`) and filter `ContainerAppConsoleLogs_CL`:

```kql
ContainerAppConsoleLogs_CL
| where TimeGenerated > ago(1h)
| where Log_s contains "Enrichment" or Log_s contains "Triage" or Log_s contains "set_priority" or Log_s contains "GSOC"
| project TimeGenerated, Log_s
| order by TimeGenerated desc
```

Look for:
- `set_priority(KEY) ‚Üí <priority>`
- `GSOC assign skipped ‚Äî JIRA_GSOC_ACCOUNT_ID not set` (or `Assigned KEY to account <id>`)
- `triage_priority(KEY): recommended=<priority>, confidence=<float>, rationale=<text>`
- `Triage override accepted: <baseline> ‚Üí <recommended>` (or `Triage override rejected: confidence below threshold`)

## 6. Rollback

If Phase 1 needs to be reverted:

```bash
az containerapp update \
  --name soc-platform --resource-group rg-soc-platform \
  --image socplatformreg.azurecr.io/soc-platform:pre-phase-1-2026-06-08 \
  --revision-suffix "rollback-$(date +%s)"
```

This restores the exact container image that was running before Phase 1 deploy. New tickets created after rollback go back to the pre-Phase-1 behaviour. Existing tickets keep whatever priority/label state they were last left in.

If config-only revert is needed (keep new code, revert label changes), set:
```
JIRA_TRIAGE_MALICIOUS_LABEL=IOC_Detection
JIRA_TRIAGE_CLEAN_LABEL=investigating
JIRA_LABEL_REMOVE_ON_CLOSE=IOC_Detection
```

## 7. Known limitations / what's next

Phase 1 ships the verdict labels and the LLM-priority-override logic. It does **not** yet:

- Surface the LLM's `rationale` in the enrichment comment (only logs it) ‚Äî added in Phase 6 (Recommendation Synthesis)
- Map alerts to MITRE ATT&CK TTPs ‚Äî Phase 2
- Look up historical alerts of the same type ‚Äî Phase 3
- Use Confluence (Customer Escalation Matrix, HRT/HVT, Asset Inventory) for context ‚Äî Phase 4
- Run Sentinel KQL queries for corroboration ‚Äî Phase 5
- Auto-close `Benign-Positive` tickets ‚Äî Phase 7

See the [roadmap](L1-TRIAGE-REDESIGN-ROADMAP.md) for the full sequence.
